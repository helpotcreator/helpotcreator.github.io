# 2021.12.11. log4j 취약점 및 패치 방법

KISA에서 긴급 공지한, 취약점 내용과 패치 방법을 소개한다.

# 개요

대한민국의 관공서 및 기업들은 백엔드를 JAVA로 운영하는 곳이 대부분이면서, log4j를 이용하는 곳이 굉장히 많을 것으로 예상되니, 해당 취약점은 대한민국 전체적으로 큰 타격을 줄 수 있기에 빠른 패치가 필요해 보인다.

  • KISA 공지open in new window
  • 아파치 재단 공지open in new window
  • Spring 재단 공지open in new window

실제로 어떻게 패치 진행하면 되는지 상세히 남기니, 개발자 및 데브옵스 분들께 도움이 되면 좋겠다.

  • 영향을 받는 버전: 2.0-beta9 ~ 2.14.1
  • 해결방안
    • 2.0-beta9 ~ 2.10.0 버전일 경우
      • JndLookup 클래스를 경로에서 제거 : zip -q -d log4j-core-*.jar org/apache/logging/log4j/core/lookup/JndiLookup.class
      • 2.15.0 버전으로 업그레이드
    • 2.10 ~ 2.14.1 버전일 경우
      • log4j2.formatMsgNoLookups 또는 LOG4J_FORMAT_MSG_NO_LOOKUPS 환경변수를 true로 설정
      • 2.15.0 버전으로 업그레이드
  • KISA에 문의: 118 전화
  • 참조
    • https://logging.apache.org/log4j/2.x/security.html
    • https://cve.mitre.org/cgi-bin/cvename.cgi?name=CVE-2021-44228
    • https://logging.apache.org/log4j/2.x/download.html

WARNING

여기서 아주 중요한 점은, 만약 log4j 버전을 2.15.0으로 업그레이드를 진행할 경우, JDK 버전이 1.8 이상이어야 한다. 따라서, log4j 버전 업그레이드가 어려운 상황이라면, logback 라이브러리로 바꾸는 것도 새로운 방안이 된다.

# log4j 관련 정보

JAVA에서 가장 유명하게 사용되는 로깅 라이브러리이다. 나는 주로 사용하는 SpringBoot 프레임웍에서는 logback 라이브러리가 기본으로 포함되어 있다.

# log4j-core 사용 여부 및 버전 확인_1

우선, 첫 번째로 gradle이나 maven의 dependencies를 확인해야 한다. maven일 경우, pom.xml 파일을 열어서 dependencies 내용 중에 log4j 관련 라이브러리가 적혀 있는지 확인한다. 특별히, log4j 중에서도 log4j-core가 패치의 대상이니, 이를 검색해서 찾아야 한다. 아래와 같은 형태이다.

<dependency>
    <groupId>org.apache.logging.log4j</groupId>
    <artifactId>log4j-core</artifactId>
    <version>2.11.0</version>
</dependency>

TIP

version 부분이 생략되어 있을 수 있다. 그럴 경우엔, 개발 시점 기준으로 프레임워크 버전과 가장 호환이 잘 되는 log4j 최신 버전이 자동으로 선택되어 사용되고 있을 것이기에, 다른 방법으로도 버전을 확인해봐야 한다.

TIP

logback이나 log4j2를 이용하는 경우도, 확인하는게 좋다. 해당 라이브러리들도 log4j를 기반으로 만들어졌기 때문이며, maven이나 gradle과 같은 빌드툴 외에 수동으로 라이브러리가 추가 되어 있을 수도 있기 때문이다.

# log4j-core 사용 여부 및 버전 확인_2

개발자나 데브옵스의 경우, 소스 파일이 노트북에 존재할 것이다. 특별히, 담당 개발자라면 maven 또는 gradle 인스톨까지 되어 있을테니, log4j-core 라이브러리 파일 자체가 노트북에 다운로드 되어 있을 것이다. 라이브러리 파일은 .jar 형태이며, 파일명에 버전이 적혀 있다. 본인은 테스트를 위해, 일부러 설치했다. 파일명을 통해 2.10.0 버전임을 확인하였다.

위 화면에서 보이듯이, 당연히 IDE에서는 라이브러리 내부의 특정 .class 파일을 직접 지울 순 없다.

# 패치 방법_1(2.0-beta9 ~ 2.10.0)

감사하게도 KISA에서 패치 방법을 공지하였지만, 약간만 더 친절하게 적어주었으면 개발자들이 속 편할 듯 하다. 나의 경우, 윈도우OS 및 gradle을 이용중인데, gradle은 라이브러리 파일을 아래에 다운받아서 관리한다. C:\Users\helpotcreator\.gradle\caches\modules-2\files-2.1\org.apache.logging.log4j\log4j-core\2.10.0

총 3개의 폴더가 나오는데, 첫 번째 폴더와 세 번째 폴더에 각각 log4j-core-2.10.0-sources.jar 및 log4j-core-2.10.0.jar파일이 존재함을 확인했다.

DANGER

아래 작업을 진행하기 전에, .jar 파일 백업은 필수이다.

본인은 해당 2개 .jar 파일을 반디집을 이용하여 열었다.

그리곤, .jar 내부의 취약점이 발견된 .class 파일을 삭제했다. org/apache/logging/log4j/core/lookup/JndiLookup.class

TIP

KISA 가이드 처럼, 커맨드에서 zip 명령어로 실행을 원할 경우 주의할 점이 있다. zip 명령어는 기본 명령어가 아니기 때문에 따로 설치가 필요하다. zip.exe 및 unzip.exe 파일을 압축하여 아래 첨부한다.

  • zipOfDOS.zip

그리고, 위 파일들을 C:\Windows\System32으로 옮긴 이후, CMD를 재 실행하면 zip 명령어 사용이 가능하다.

# 패치 방법_2(2.10.0 ~ 2.14.1)

log4j-core-2.10 버전의 경우는, KISA 가이드를 보니, 방법1 또는 방법2 무엇으로 진행해도 되는 것으로 표기되어 있다. 본인의 경우는 이미 위 패치 방법_1을 통해 진행했었기 때문에 이번에는 2.14.1 버전의 log4j를 설치해서 진행해보겠다.

KISA 가이드 내용만으로는 어떻게 진행해야할지 모르겠다.

좀 더 찾아보니, Spring 재단 가이드에 해당 방법이 나와 있다.

너무 쉬운 방법이다. log4j-core-*.jar 라이브러리 파일을 대상으로 패치를 하는것이 아니라, log4j를 이용하여 개발 완료한 자신의 프로젝트 .jar 파일을 대상으로 아래 구문을 실행만 해주면 된다. 물론, 아래 구문에서 myapp.jar 부분은 자신의 프로젝트 파일명으로 변경하여 진행한다.

java -Dlog4j2.formatMsgNoLookups=true -jar myapp.jar
